# 碑文金石资料检索系统 - 架构设计文档 (v2.0)

## 1. 系统概览 (System Overview)
本系统旨在为金石碑文资料提供数字化检索、管理与展示服务。v2.0 版本在基础检索功能之上，新增了**增量数据导入**与**冲突处理**机制，支持通过 Word 文档批量或单条更新数据，并确保数据的一致性与完整性。

## 2. 核心架构设计 (Core Architecture)

### 2.1 模块划分
系统采用 **分层架构 (Layered Architecture)**，模块职责清晰分离：

```mermaid
graph TD
    Client[前端 Client (Browser)] --> API[API 层 (FastAPI)]
    API --> Logic[业务逻辑层 (CRUD / Parser)]
    Logic --> Data[数据访问层 (SQLAlchemy)]
    Data --> DB[(SQLite 数据库)]
    Logic --> FS[文件系统 (Static Images)]
```

- **API 层 (`app/main.py`)**: 
  - 负责接收 HTTP 请求（搜索、上传、冲突处理）。
  - 路由分发与参数校验。
- **业务逻辑层**:
  - **检索逻辑 (`app/crud.py`)**: 负责带权重的模糊搜索、字段过滤。
  - **解析逻辑 (`scripts/word_parser.py`)**: 负责 Word 文档的解析、图片提取、HTML 转换。**此模块被设计为独立可复用组件**，既可由 CLI 脚本调用，也可由 API 调用。
- **数据访问层 (`app/models.py`, `app/database.py`)**: 定义数据模型与数据库交互。
- **基础设施**: Docker 容器化部署，通过 Volume 挂载实现数据持久化。

### 2.2 数据流设计 (Data Flow)

#### 2.2.1 搜索流程
1. 用户输入关键词 -> API `/api/search`
2. CRUD 模块构建动态 SQL 查询 (Weighted Search)
3. 返回 JSON 结果 (包含图片 URL 列表)

#### 2.2.2 导入流程 (新增核心)
1. **上传**: 用户通过 API `/api/upload` 上传 Word 文件。
2. **解析**: 后端调用 `word_parser.parse_docx(file)`，在内存中生成结构化数据列表（不直接入库）。
3. **查重**: 遍历解析结果，以 **器名 (Name)** 为唯一键查询数据库。
   - **不存在**: 标记为 `NEW` -> 写入数据库，保存图片至文件系统。
   - **存在**: 标记为 `CONFLICT` -> **跳过写入**，记录冲突详情。
4. **反馈**: API 返回 JSON 报告 `{success_count: 10, skipped_count: 2, skipped_items: [...]}`。

## 3. 详细设计 (Detailed Design)

### 3.1 数据库设计 (Schema)
表名: `inscriptions`

| 字段名 | 类型 | 索引 | 唯一性 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | Integer | PK | YES | 自增主键 |

| `image_url` | Text | NO | NO | 图片路径 (JSON 列表) |
| ... | ... | ... | ... | 其他元数据 (时代, 出土等) |

> **变更点**: `name` 字段增加 `unique=True` 约束，作为业务主键。

### 3.2 接口设计 (API Definitions)

#### 3.2.1 资料上传接口
- **URL**: `POST /api/upload`
- **Content-Type**: `multipart/form-data`
- **参数**: `file` (Word 文档)
- **响应**:
```json

  "skipped": 2,
```

## 4. 关键技术决策 (Key Decisions)

1.  **解析逻辑解耦**: 将 `word_parser.py` 中的解析逻辑从“解析即保存”重构为“解析返回对象”。这使得 API 可以先获取解析结果，进行查重逻辑判断后，再决定是否保存。
2.  **后置报告模式**: 采用“先处理，后报告”的策略。相比于“预检查”模式，这种方式减少了文件读取次数，且对于绝大多数非冲突数据处理效率更高。
3.  **图片处理**: Word 中的行内小图（如金文）被转换为 `<img class="inline-char">` 标签嵌入在释文 HTML 中；大图则被提取为附件。这种处理逻辑在导入时保持不变。
